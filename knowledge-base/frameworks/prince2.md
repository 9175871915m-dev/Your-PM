# PRINCE2 — Projects in Controlled Environments

## Overview
PRINCE2 is a structured, process-based project management method developed by AXELOS. It is the dominant framework in UK public sector and widely used across Europe, Australia, and the Middle East.

## 7 PRINCE2 Principles
1. **Continued Business Justification** — project must always have a valid business case
2. **Learn from Experience** — lessons captured and applied
3. **Defined Roles and Responsibilities** — everyone knows their role
4. **Manage by Stages** — projects divided into management stages
5. **Manage by Exception** — delegate authority within agreed tolerances
6. **Focus on Products** — define what will be delivered, not just activities
7. **Tailor to Suit** — adapt the method to the project context

## PRINCE2 Themes
| Theme | Purpose |
|---|---|
| Business Case | Justification for the project |
| Organisation | Roles and responsibilities |
| Quality | Standards, quality criteria, acceptance criteria |
| Plans | Project Plan, Stage Plans, Team Plans |
| Risk | Risk Register, threat/opportunity management |
| Change | Issue Register, Configuration Management |
| Progress | Monitoring against Stage Plan |

## PRINCE2 Processes
1. **Starting Up a Project (SU)** — appoint PM, produce Project Brief
2. **Initiating a Project (IP)** — produce Project Initiation Documentation (PID)
3. **Directing a Project (DP)** — Project Board governance throughout
4. **Controlling a Stage (CS)** — PM day-to-day management
5. **Managing Product Delivery (MP)** — Team Manager delivers work packages
6. **Managing a Stage Boundary (SB)** — plan next stage, update Business Case
7. **Closing a Project (CP)** — handover, lessons learned, formal closure

## PRINCE2 Organisation Structure
- **Project Board** (above the PM)
  - **Executive** — owns the Business Case, accountable for project
  - **Senior User** — represents users of the product
  - **Senior Supplier** — represents those delivering the product
- **Project Manager** — day-to-day management
- **Team Manager** — manages specialist delivery teams
- **Project Assurance** — independent oversight for Board
- **Project Support** — admin and PMO support

## PRINCE2 Documents
| Document | Purpose |
|---|---|
| Project Mandate | Initial trigger from corporate body |
| Project Brief | High-level project description |
| PID | Full project definition (scope, plan, business case) |
| Business Case | Justification — costs, benefits, risks, timescales |
| Risk Register | All risks — threat and opportunity |
| Issue Register | All change requests and problems |
| Quality Register | Quality activities planned and completed |
| Lessons Log | Lessons captured during delivery |
| Work Package | Authorisation for team to deliver a product |
| Highlight Report | Regular progress report to Project Board |
| Exception Report | When tolerance is forecast to be exceeded |
| End Stage Report | Summary at end of each management stage |
| End Project Report | Final report at project closure |

## PRINCE2 vs PMBOK
| Factor | PRINCE2 | PMBOK |
|---|---|---|
| Origin | UK / AXELOS | USA / PMI |
| Type | Method (prescriptive) | Body of Knowledge (guidance) |
| Focus | Process and governance | Knowledge areas and tools |
| Best for | UK public sector, formal governance | International, flexible application |
| Scalability | Tailoring built in | Tailoring described |
| Certification | Foundation + Practitioner | CAPM + PMP |

## When to Use PRINCE2
- Government, NHS, MOD, local authority projects
- Projects requiring strong governance and reporting
- Multi-organisation projects with complex stakeholders
- When Business Case justification tracking is critical
